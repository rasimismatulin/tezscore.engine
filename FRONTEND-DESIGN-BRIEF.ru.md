# TezScore Engine — Дизайн-бриф фронтенда (для дизайнера)

> Документ для UI/UX-дизайнера. Описывает, как работает система, что делают пользователи, из каких экранов состоит приложение, какие нужны меню, действия, формы, модальные окна, гриды и компоненты. Цель — дать дизайнеру полную картину, чтобы отрисовать готовый макет.
>
> Термины системы (`Analyzer / Score Model`, `Condition`, `Function`, `Rule`, `DSL`, `ContextType` и т.д.) намеренно оставлены на английском — это ключевые сущности продукта, они будут так же называться в интерфейсе и коде.
>
> Технологии фронтенда (для контекста): Nuxt 4 + Vue 3.5 + Tailwind, тёмная/светлая тема (color-mode), иконки lucide, мультиязычность (i18n). Все wireframe ниже — схематичные (ASCII), дизайнер превращает их в hi-fi макет.

## 1. О продукте (коротко)

TezScore Engine — платформа **кредитного скоринга**. Кредитные специалисты **сами настраивают** логику принятия решения по кредиту, без программистов. Система берёт данные клиента (пока — кредитную историю) и возвращает один результат-«зону»: например **Yaşıl** (зелёный — одобрить), **Sarı** (жёлтый — на рассмотрение специалиста), **Narıncı** (оранжевый), **Qırmızı** (красный — отказ).

Логика собирается из трёх переиспользуемых «кирпичиков»:

- **Function** — небольшая вычислительная операция над данными (посчитать, найти, отфильтровать).
- **Condition** — именованное условие, которое возвращает «да/нет» (использует Functions и данные).
- **Rule** — правило вида «ЕСЛИ (условия) ТО (результат)». Набор Rules внутри Score Model даёт итоговую зону.

Настраивают всё это через **Formula DSL** — простой текстовый язык формул (с редактором, автодополнением и подсветкой ошибок).

Ключевая идея интерфейса: специалист должен **видеть результат сразу** — есть режим **Live Preview**, где формула вычисляется в реальном времени на реальных или тестовых данных.

## 2. Глоссарий (обязательно к пониманию дизайнером)

| Термин | Что это | Как показывать в UI |
|---|---|---|
| **Score Model (Analyzer)** | Настроенная модель скоринга: список результатов + упорядоченные Rules. Верхнеуровневая сущность. | Главная сущность. Список + карточка-редактор. |
| **Result (зона)** | Возможный итог модели: название + код + цвет (Yaşıl/Sarı/Narıncı/Qırmızı). | Цветные бейджи. |
| **Rule** | Строка «WHEN <условие> THEN <результат>». Правила упорядочены, срабатывает первое подходящее. | Строки конструктора правил. |
| **OTHERWISE** | Обязательный результат «по умолчанию», если ни одно правило не сработало. | Зафиксированная строка внизу списка правил. |
| **Condition** | Переиспользуемое условие (да/нет). Может иметь **параметры**. | Список + редактор + чипы `@Имя` в правилах. |
| **Function** | Переиспользуемая операция над массивами/числами. Имеет параметры и тип результата. | Список + редактор. |
| **DSL** | Язык формул для Functions/Conditions/Rules. | Текстовый редактор с автодополнением. |
| **ContextType** | Тип входных данных: `Credit` (сейчас), в будущем `Personal`, `Employee`, `Pension`. | Селектор/фильтр; влияет на доступные параметры. |
| **FIN** | Идентификатор клиента (по нему тянется кредитная история). | Поле ввода в тесте/preview. |
| **Liability** | Отдельный кредит (обязательство) клиента. | Строки в тестовых данных, «источник» в формулах. |
| **Payment Timeline** | Массив просрочек по месяцам за 24 месяца. Индекс 0 = текущий месяц; `null` = нет данных (кредит закрыт/не начат). | Сетка месяцев (grid) со значениями и пустыми ячейками. |
| **Sample Dataset** | Сохранённый тестовый набор данных клиента (dummy `CreditHistoryModel`) для preview. | Список + билдер. |
| **Trace** | Пошаговый разбор: какое правило сработало и почему. | «Водопад» результатов теста. |
| **Draft / Published / Version** | Черновик → публикация (с проверкой) → зафиксированная версия. | Статус-бейджи, кнопка Publish. |

## 3. Роли и доступ

Пока три роли (права детализируем позже, дизайн должен предусмотреть скрытие/блокировку действий по роли):

| Роль | Что может (ориентир) |
|---|---|
| **Admin** | Всё: настройка моделей, пользователи, активация/удаление. |
| **Manager** | Создавать/редактировать Score Models, Conditions, Functions; тестировать. |
| **User** | Просмотр, запуск тестов (ограниченно). |

UI-требование: действия, недоступные роли, показываются **заблокированными с подсказкой** или скрываются. Дизайнеру нужны состояния кнопок «disabled + tooltip».

## 6. Аутентификация

### 6.1 Login
Вход по **номеру телефона (username)** и паролю.

```
        ┌──────────────────────────────┐
        │           TezScore           │
        │      Вход в систему          │
        │                              │
        │  Номер телефона              │
        │  ┌────────────────────────┐  │
        │  │ 994 50 XXX XX XX        │  │
        │  └────────────────────────┘  │
        │  Пароль                      │
        │  ┌────────────────────────┐  │
        │  │ ••••••••            👁  │  │
        │  └────────────────────────┘  │
        │                              │
        │  [      Войти          ]     │
        │                              │
        └──────────────────────────────┘
```

- Валидация телефона: формат `994XXZZZZZZZ` (XX — код оператора 50/51/55/70/77/10/99).
- Ошибки: неверный логин/пароль — общий текст (без утечки, какой из полей неверный).

### 6.2 OTP (второй фактор)
После логина приходит **4-значный** код по SMS. В dev-режиме код всегда `0123` — нужен **баннер-подсказка** «Тестовый режим: код 0123».

```
        ┌──────────────────────────────┐
        │   Подтверждение входа        │
        │   Код отправлен на 994•••••53 │
        │                              │
        │     ┌──┐ ┌──┐ ┌──┐ ┌──┐      │
        │     │ 0│ │ 1│ │ 2│ │ 3│      │   ← 4 отдельные ячейки
        │     └──┘ └──┘ └──┘ └──┘      │
        │                              │
        │   Повторить код через 02:59  │   ← таймер TTL 3 мин
        │   [   Подтвердить   ]        │
        │                              │
        │  ⓘ Тестовый режим: код 0123  │   ← только dev
        └──────────────────────────────┘
```

- Таймер обратного отсчёта (3 мин), кнопка «Отправить повторно» после истечения.
- Блокировка после **5 неверных попыток** — показать сообщение об ошибке и предложить повторить позже.

## 8. Rule Builder (конструктор правил) — ключевой экран

Назначение: собрать упорядоченный набор правил. Логика: **сверху вниз, срабатывает первое подходящее**; в самом низу — обязательный **OTHERWISE**.

```
┌ Rules ────────────────────────────────────────────────────  [+ Правило] ─┐
│                                                                            │
│  1  ≡  WHEN  [ @Any90Plus OR @Overdue30PlusTotal        ]  THEN [🔴 RED ▾] │
│  2  ≡  WHEN  [ @Tables7or8                              ]  THEN [🟠 ORANGE▾]│
│  3  ≡  WHEN  [ @Tables5or6                              ]  THEN [🟡 YELLOW▾]│
│  4  ≡  WHEN  [ @GreenSingle3M OR @EmptyHistory          ]  THEN [🟢 GREEN ▾]│
│ ───────────────────────────────────────────────────────────────────────── │
│  ⛝  OTHERWISE                                             THEN [🔴 RED ▾]  │  ← нельзя удалить
└────────────────────────────────────────────────────────────────────────┘
```

Для каждого правила:
- **Порядковый номер** + drag-and-drop (`≡`) для изменения порядка.
- **Поле условия (WHEN):** это DSL-выражение. Ввод двумя способами (переключатель):
  - **Режим «чипы»** — выбор готовых Conditions как тегов `@Имя`, соединённых `AND/OR/NOT` и скобками (визуальный сборщик).
  - **Режим «формула»** — прямой ввод DSL с автодополнением (см. раздел 11).
- **Селектор результата (THEN):** выпадающий список Results этой модели (цветные).
- Кнопки: дублировать правило, удалить, комментарий к правилу.
- **Валидация в реальном времени:** подсветка ошибки в условии, значок статуса (✓/✕) у каждого правила.

**Важно для дизайна:** визуально показать, что порядок важен (нумерация, «первое совпадение выигрывает»), и что OTHERWISE — зафиксирован (нельзя удалить/переместить), это гарантия «всегда есть результат».

**Опциональный визуальный редактор условия (Condition-конструктор), модалка:**

```
┌─ Условие правила #3 ──────────────────────────────────────────────┐
│  [ @Tables5or6 ]  ( AND ▾ )  [ NOT ]  [ @Recovered3M ]            │
│  [+ добавить Condition]  [+ группа ( ) ]                          │
│  ───────────────────────────────────────────────────────────     │
│  Предпросмотр DSL:  @Tables5or6 AND NOT @Recovered3M              │
│  Статус: ✓ корректно                                              │
│                                            [Отмена] [Применить]    │
└──────────────────────────────────────────────────────────────────┘
```

## 9. Conditions

### 9.1 Список Conditions (Grid)
Глобальная библиотека условий (общие для всех моделей).

```
┌ Conditions ───────────────────────────────────────────────  [+ Создать] ┐
│ Поиск [____]  ContextType [Credit ▾]                                      │
├──────────────────────────────────────────────────────────────────────────┤
│ Name                         │ Code            │ Параметры │ Использ. │ ⋯  │
│──────────────────────────────┼─────────────────┼───────────┼──────────┼────│
│ Нет просрочки за 3 мес.      │ NoOverdueLast3M │ src:Matrix│ 4 модели │ ⋯  │
│ Есть 90+ дней просрочки      │ Any90Plus       │ —         │ 2 модели │ ⋯  │
│ Пустая история               │ EmptyHistory    │ —         │ 6 моделей│ ⋯  │
└──────────────────────────────────────────────────────────────────────────┘
```

- Колонки: Name (человекочитаемое, показывается в Rule Builder), Code, параметры (сигнатура), «где используется» (счётчик), меню.
- **Delete guard:** если Condition используется в активной модели — удаление блокируется (показать где используется).

### 9.2 Редактор Condition (двухпанельный: настройка + Live Preview)

```
┌ Condition: NoOverdueLast3M ─────────────────────────────┬─ Live Preview ──────┐
│ Name*   [Нет просрочки за 3 мес.        ]               │ Источник данных:     │
│ Code*   [NoOverdueLast3M                ]               │ ( ) FIN  (•) Sample  │
│ Context [Credit ▾]                                      │ [Клиент №1 (демо) ▾] │
│ Описание[__________________________________]            │                      │
│                                                         │ Результат:           │
│ Параметры:                              [+ параметр]    │   ┌──────────────┐   │
│   src : Matrix        [✎][🗑]                            │   │  ✓ TRUE       │   │
│                                                         │   └──────────────┘   │
│ Выражение (DSL):                                        │                      │
│ ┌─────────────────────────────────────────────────┐    │ Разбор:              │
│ │ Max(SubArray(MergeMax(src), 0, 2)) <= 10         │    │  MergeMax(src)=[..]  │
│ │                                                  │    │  Max(0..2)=8         │
│ └─────────────────────────────────────────────────┘    │  8 <= 10 → TRUE      │
│ Статус: ✓ выражение корректно (тип: Boolean)            │                      │
│                                        [Отмена][Сохр.]  │                      │
└─────────────────────────────────────────────────────────┴──────────────────────┘
```

- **Параметры** (как у Function): имя + тип (`Number/Series/Matrix/Boolean/Predicate`). Модалка добавления параметра.
- **DSL-редактор** с автодополнением и live-валидацией; под ним — статус и определённый тип результата (для Condition всегда `Boolean`).
- **Live Preview** справа — сразу видно `TRUE/FALSE` и разбор вычисления на выбранных данных.

## 10. Functions

### 10.1 Список Functions (Grid)

```
┌ Functions ────────────────────────────────────────────────  [+ Создать] ┐
│ Name              │ Code        │ Сигнатура                    │ Тип  │ ⋯ │
│───────────────────┼─────────────┼──────────────────────────────┼──────┼───│
│ Макс. за окно     │ WindowMax   │ (s:Series, from:N, to:N)     │ N    │ ⋯ │
│ Чистые месяцы     │ CleanSince  │ (s:Series, idx:N)            │ N    │ ⋯ │
└──────────────────────────────────────────────────────────────────────────┘
```

- Колонки: Name, Code, сигнатура (параметры), тип результата, «где используется», меню.
- Есть встроенные (built-in) функции (`Count`, `IndexOf`, `SubArray`, `MergeMax`, `TrimStart`, …) — они **не редактируются**, но видны в справочнике/автодополнении. Стоит показать раздел «Встроенные функции» отдельной вкладкой или фильтром «Только пользовательские / Все».

### 10.2 Редактор Function (двухпанельный, аналогично Condition)

```
┌ Function: WindowMax ────────────────────────────────────┬─ Live Preview ──────┐
│ Name*  [Макс. за окно            ]                      │ Источник: [Sample ▾] │
│ Code*  [WindowMax                ]                      │ Аргументы для теста: │
│ Тип результата [Number ▾]                               │  s = [0,10,5,null…]  │
│ Параметры:                              [+ параметр]    │  from = 0            │
│   s : Series   [✎][🗑]                                   │  to = 2              │
│   from : Number[✎][🗑]                                   │                      │
│   to : Number  [✎][🗑]                                   │ Результат: 10        │
│ Тело (DSL):                                             │                      │
│ ┌─────────────────────────────────────────────────┐    │                      │
│ │ Max(SubArray(s, from, to))                       │    │                      │
│ └─────────────────────────────────────────────────┘    │                      │
│ Статус: ✓ корректно                    [Отмена][Сохр.]  │                      │
└─────────────────────────────────────────────────────────┴──────────────────────┘
```

- Для preview функции нужен блок **ручного ввода аргументов** (по сигнатуре): числа, массивы (Series), матрицы.

## 11. DSL Editor — общий компонент (используется в Rules/Conditions/Functions)

Самый важный интерактивный компонент. Требования к дизайну:

- **Однострочный и многострочный** режимы (условия правил — короткие; тело функции — может быть многострочным).
- **Автодополнение** (dropdown при вводе): список Direct Parameters, источников (`ActiveLiabilities`, `ClosedLiabilities`…), built-in и пользовательских Functions (с сигнатурой), Conditions (`@Code`). Показывать тип и краткое описание.
- **Подсветка синтаксиса:** ключевые слова (`WHEN/THEN/OTHERWISE/where/AND/OR/NOT`), функции, `@условия`, числа, строки.
- **Инлайновые ошибки:** волнистое подчёркивание + текст ошибки с позицией (строка/столбец). Пример: `= null` → ошибка «используйте IsNull(x)».
- **Статус-строка под редактором:** ✓ корректно (тип: Boolean) / ✕ ошибка.
- **Панель-справочник** (сбоку или по кнопке `ⓘ`): каталог функций с примерами.

```
┌ DSL ──────────────────────────────────────────────── ⓘ Справочник ┐
│ Count(ActiveLiabilities, where dueDays > 10) >= 2                    │
│                                   ▲                                  │
│        ┌─────────────────────────────────────────┐                  │
│        │ dueDays      (Number) — просрочка месяца │  ← автодополнение│
│        │ DataMonthCount(s)  (Number)              │                  │
│        │ @NoOverdueLast3M(src)  (Boolean)         │                  │
│        └─────────────────────────────────────────┘                  │
│ ✓ тип: Boolean                                                       │
└──────────────────────────────────────────────────────────────────────┘
```

## 12. Live Preview (панель предпросмотра)

Появляется в редакторах Conditions/Functions/Rules. Показывает результат **в реальном времени** (пересчёт при вводе, с задержкой ~debounce).

**Селектор источника данных (3 варианта):**

```
┌ Источник данных ───────────────────────────────┐
│ (•) По FIN     ( ) Вручную     ( ) Sample       │
│ ┌───────────────────────────────────────────┐  │
│ │ FIN: [ 1234567 ]              [Загрузить]  │  │
│ └───────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

**Рендер результата зависит от типа выражения:**
- **Boolean** → крупный бейдж TRUE/FALSE + разбор (какие под-Conditions сработали).
- **Number** → число.
- **Series** → сетка месяцев (см. ниже), с подсветкой ячеек, попавших под условие, и серыми `null`-ячейками.
- **Matrix** → несколько строк-Series (по одному кредиту).

**Отрисовка Series / Payment Timeline (важно):**

```
 мес:  0    1    2    3    4    5   …  (0 = текущий)
      ┌───┬───┬───┬───┬───┬───┐
      │ 0 │10 │ 5 │ — │ — │30 │ …   ( — = null, нет данных)
      └───┴───┴───┴───┴───┴───┘
        ▲выделены месяцы, где dueDays>10
```

## 14. Тестирование модели (Test by FIN) + Trace

Вкладка «Test / Preview» внутри Score Model. Прогнать модель на клиенте и увидеть итог + разбор.

```
┌ Test / Preview ───────────────────────────────────────────────────────────┐

│ FIN [ 1234567 ]                                             [ Запустить ]   │

│ Сработало правило #2: @Tables7or8                                          │
├────────────────────────────────────────────────────────────────────────────┤
│ Trace (разбор):                                                            │
```

- **Data Grid / таблица:** сортировка колонок, пагинация (или бесконечная прокрутка), поиск, фильтры, «действия в строке» (⋯), выбор строк (при необходимости), пустое состояние.

- **Формы:** метки полей, обязательные поля (*), inline-ошибки под полем, состояния disabled, подсказки (tooltips).
- **Тосты/нотификации:** успех/ошибка/инфо (сохранено, опубликовано, ошибка сервера).
- **Бейджи статусов:** Draft/Published/Inactive; зоны результатов (цвета).
- **Chips/теги:** для `@Conditions` в правилах.
- **Селектор источника данных** (FIN/Manual/Sample) — переиспользуется в preview и тесте.
- **Диалог «Несохранённые изменения»** при попытке уйти со страницы редактора.

## 20. Что важно НЕ упустить (резюме для дизайнера)

- Порядок правил и «первое совпадение выигрывает» + зафиксированный **OTHERWISE** — это нужно **показать визуально**.
- **Live Preview** — сердце UX: специалист постоянно видит результат формулы. Панель предпросмотра должна быть удобной и всегда под рукой в редакторах.
- **Series / сетка месяцев** (24 месяца, `null` = нет данных, индекс 0 = текущий) — встречается и в preview, и в билдере данных; продумать компактный и понятный вид.
- **Trace** должен объяснять решение «человеческим» образом (какое правило, почему, с числами).
- Роли ограничивают действия — заложить disabled/hidden состояния.
- Черновик → **Publish** (с проверкой) → версия; несохранённые изменения предупреждать.
